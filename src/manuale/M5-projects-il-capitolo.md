# M5 — I Projects: il capitolo operativo

**Codice:** M5 · **Versione:** 0.1 · 13/07/2026 · **Estende:** F4 §§5–6 con il ciclo di vita completo e l'uso per ruolo

## 1. Il ciclo di vita del Project evento

1. **Nascita** (all'avvio dell'evento): istruzioni dal template (→ F4 §5.2), caricamento di scheda dati e programma vigente, condivisione al team ("può usare"; "può modificare" solo owner e vice).
2. **Regime**: la conoscenza si aggiorna a ogni versione ufficiale del programma (sostituzione, non aggiunta); i risultati validati si depositano; le chat si rinominano (→ M4 §2).
3. **Picco** (ultime 6 settimane): revisione quindicinale della conoscenza — 10 minuti dell'owner: c'è qualcosa di scaduto? manca l'ultima versione di qualcosa?
4. **Chiusura**: si depositano i materiali "lezioni apprese" (cosa rifare uguale/diverso), i template nati durante l'evento migrano alla libreria (→ M6), poi **si archivia** — ordine, e le condivisioni si azzerano [FATTO].

## 2. Le istruzioni dei Project trasversali (esempi pronti)

**Amministrazione eventi**
```
Sei l'assistente dell'amministrazione di B Best. Contesto: gestione
amministrativa di eventi ECM; fatturazione con Fattureincloud. Tono
professionale e asciutto; italiano. Regole: importi, scadenze e riferimenti
di fattura vanno sempre marcati "da verificare sul gestionale" se non
forniti nel messaggio; mai completare IBAN o dati di pagamento; le
comunicazioni a sponsor citano sempre il riferimento contratto [RIF].
```

**Comunicazione**
```
Sei l'assistente della comunicazione di B Best/HERA per eventi medici ECM.
Tono: professionale, chiaro, mai sensazionalista; italiano (inglese su
richiesta). Regole: ogni claim scientifico deve avere fonte o essere
marcato "da verificare"; niente promesse su crediti ECM non ancora
accreditati; nomi di relatori solo se confermati; massimo 3 hashtag.
```

## 3. L'uso per ruolo (dentro il Project evento)

| Ruolo | Chat tipiche | Prompt ricorrente da libreria |
|---|---|---|
| Eventi | fornitori, faculty-logistica, venue | Sollecito deliverable · richiesta preventivo |
| Amministrazione | preventivi, solleciti pagamento | Promemoria fattura [RIF][SCADENZA] |
| Segreteria scientifica | programma, quiz, corrispondenza EN | Bozze quiz ECM (con dicitura di validazione) |
| Comunicazione | annunci, newsletter, post | Annuncio in N versioni per canale |

## 4. Governance minima (mezza pagina, non un regolamento)

- **Owner**: decide cosa entra nella conoscenza, fa la revisione quindicinale, archivia a fine evento, risponde di ciò che il Project condiviso contiene (→ F6 §7 punto 6).
- **Chi può modificare**: owner + un vice. Tutti gli altri "può usare".
- **Naming**: `Romanestesia 2026` per gli eventi; i trasversali senza anno.
- **Un dubbio ricorrente**: "creo un Project per questo?" — test: *ha vita oltre le due settimane E materiale proprio?* Se no, è una chat.
